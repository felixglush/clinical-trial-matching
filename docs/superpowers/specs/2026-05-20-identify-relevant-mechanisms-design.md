# identify-relevant-mechanisms: Design

**Date:** 2026-05-20
**Status:** Draft (pending user review)
**Scope:** Implementation of the `identify-relevant-mechanisms` LangGraph node and its supporting tooling. Out of scope: `find-repurposing-candidates`, `mechanism-plausibility` (subgraph), `generate-search-strategy`. Those consume `state.mechanisms` but are stubbed elsewhere.

## Goal

For each active condition on a patient, query PrimeKG (Neo4j) for the associated genes and biological-process pathways, then have an LLM rank-and-filter the candidate mechanisms to the top few most clinically relevant for trial matching. Write the result to `state.mechanisms`.

This is the first node in the workflow that bridges patient data (SNOMED-coded FHIR) to the biomedical knowledge graph (MONDO-coded PrimeKG).

## Design decisions

| Decision | Choice | Rationale |
|---|---|---|
| KG access from the node | `neo4j-driver` directly via existing `apps/agent/src/tools/kg.ts` | Neo4j MCP is for ad-hoc / Studio use, not for the programmatic runtime. Determinism and testability matter on the critical path. |
| Condition → PrimeKG disease resolution | MONDO SSSOM crosswalk (SNOMED CT → MONDO) | PrimeKG diseases are MONDO-only (no SNOMED in `node_source`). Crosswalk is authoritative; the alternative (name-string matching) is fragile for granular PrimeKG subtypes. |
| Which conditions to process | All conditions with `clinicalStatus ∈ {active, recurrence, relapse}` or missing | Lets the LLM weigh comorbid drivers (CKD, T2DM) without hard-coding a "primary condition" heuristic. |
| LLM role | Rank, filter, and add per-mechanism rationale | Mechanism count is bounded; an LLM call is cheap. Rationale aids debugging and downstream prompts. Heuristic-only (skip LLM) drops the trial-context signal entirely. |
| `supportingPaths` field | Populated, capped to ≤2 paths per top pathway, constructed in TypeScript from `sharedGeneIds` returned by the pathway query | Useful for traceability / UI without a third Cypher round-trip per condition. |
| Crosswalk artifact location | Committed JSON at `apps/agent/src/data/snomed-to-primekg.json`, generated by a build script | Small (<500KB), boots offline on LangGraph Platform, no runtime SSSOM download. |

## Components

### Schema change — `packages/shared/src/mechanism.ts`

Add a required `rationale: z.string()` field to `MechanismSchema`. Verified safe: no current code constructs `Mechanism` objects; all consumers (`prompts/search-strategy.ts`, `prompts/repurposing.ts`, `prompts/mechanism-plausibility.ts`, `subgraphs/trial-eval/state.ts`) only accept `Mechanism[]` and are stubbed.

### Build-time crosswalk — `scripts/build-mondo-crosswalk.ts` (new)

Adds `pnpm kg:build-crosswalk` (with `data/kg/nodes.csv` as a prerequisite — fail with clear message if missing).

**Inputs**
- `mondo.sssom.tsv` from MONDO's GitHub release (`https://github.com/monarch-initiative/mondo/releases/latest/download/mondo.sssom.tsv`). Download to `data/kg/raw/mondo.sssom.tsv` (skip if exists, like the PrimeKG download script).
- `data/kg/nodes.csv` (already built by `kg:build-subset`).

**Algorithm**
1. Stream PrimeKG nodes; for `node_type == "disease"`, build `mondoNumericId → { primekgNodeId, primekgName }`. For `node_source == "MONDO_grouped"`, split `node_id` on `_` and map every member numeric ID to the same PrimeKG node.
2. Stream SSSOM rows. Keep only rows where `object_id` starts with `SNOMEDCT:` (the schema maps MONDO → external; SNOMED is the object) and `predicate_id` ∈ {`skos:exactMatch`, `skos:closeMatch`}. Drop `broadMatch` / `narrowMatch` — too lossy for clinical use.
3. For each kept row, parse the MONDO numeric ID from `subject_id` (strip `MONDO:` prefix and leading zeros). Look up the PrimeKG node. Skip if not in our subset.
4. Collision policy: same SNOMED code mapped to two MONDO entries → prefer `skos:exactMatch` over `skos:closeMatch`; if still tied, keep first by SSSOM row order.
5. Write `apps/agent/src/data/snomed-to-primekg.json`.

The script logs counts (SSSOM rows scanned, SNOMED entries kept, PrimeKG diseases covered) for sanity.

### Runtime — `apps/agent/src/tools/snomed-mondo.ts` (new)

```ts
import crosswalk from "../data/snomed-to-primekg.json" with { type: "json" };

export type ResolvedDisease = {
  primekgNodeId: string;
  primekgName: string;
  mondoId: string;
};

export function resolveSnomedCondition(snomedCode: string): ResolvedDisease | null {
  return crosswalk[snomedCode] ?? null;
}
```

Pure lookup. JSON loaded once at module init. No I/O at call time.

`findSharedPathways` and `pathBetween` stubs are left untouched; they belong to `find-repurposing-candidates` and `mechanism-plausibility` and are out of scope here.

**Cypher (1) — gene targets**
```cypher
MATCH (d:Node {id: $diseaseId})-[:`associated with`]-(g:Node {type: 'gene_protein'})
RETURN g.id AS id, g.name AS name
ORDER BY g.name
```
No row cap. `ORDER BY g.name` makes the result deterministic so the compact LLM prompt (which takes the first 8) is reproducible across runs. Cancer diseases typically have a few hundred associated proteins; the full list still lands in the stored `Mechanism.geneTargets`.

**Cypher (2) — pathways, ranked by shared-protein overlap**
```cypher
MATCH (d:Node {id: $diseaseId})-[:`associated with`]-(g:Node {type: 'gene_protein'})
                                -[:`interacts with`]-(p:Node {type: 'biological_process'})
WITH p, collect(DISTINCT g.id) AS sharedGeneIds
WITH p, sharedGeneIds, size(sharedGeneIds) AS shared
ORDER BY shared DESC
LIMIT $pathwayLimit
RETURN p.id AS id, p.name AS name, sharedGeneIds
```
Default `pathwayLimit = 15`.

**Constructing `supportingPaths` in TypeScript.** For each returned pathway, take up to 2 of its `sharedGeneIds`. For each gene, emit one `KGPath` with `nodes = [disease, gene, pathway]` and `edges = [{source: diseaseId, target: geneId, relation: "associated with"}, {source: geneId, target: pathwayId, relation: "interacts with"}]`. Gene/pathway names come from the query results; disease name from the resolved crosswalk entry.

### Runtime — `apps/agent/src/prompts/mechanism.ts` (modify stub)

Build the prompt + structured-output zod schema. Compact the candidates: condition name, top 8 gene names, top 8 pathway names per condition.

```ts
const PicksSchema = z.object({
  picks: z.array(z.object({
    conditionId: z.string(),
    rationale: z.string(),
  })).max(5),
});
```

Output is **ordered** by clinical priority. We honor the LLM's order when constructing `state.mechanisms`.

### Runtime — `apps/agent/src/nodes/identify-relevant-mechanisms.ts`

Orchestration only:

1. `if (!state.patientProfile) return { error: "No patient profile available" }`.
2. Filter conditions: `clinicalStatus` ∈ {`active`, `recurrence`, `relapse`} OR `clinicalStatus` is undefined.
3. Call `buildCandidateMechanisms(filtered)`.
4. If empty result, return `{ mechanisms: [] }` (not an error — see "Error handling" below).
5. Call LLM with patient summary + candidates; get `picks: { conditionId, rationale }[]`.
6. Build `Mechanism[]` by mapping `picks` to their candidates (in pick order), attaching `rationale`. Drop picks whose `conditionId` doesn't match any candidate.
7. Return `{ mechanisms }`.

## Data flow / types

```
PatientProfile.conditions[i]
  ┃ .code (SNOMED)
  ┃ .display
  ┃ .clinicalStatus
  ▼
resolveSnomedCondition(code) → ResolvedDisease | null
  ┃ .primekgNodeId
  ┃ .primekgName
  ▼
Cypher: (findGeneTargetsForDisease, findPathwaysForDisease)
  ▼
CandidateMechanism {              // internal type, no rationale
  conditionId: SNOMED code,       // for traceability back to the FHIR condition
  conditionName: condition.display,
  geneTargets: KGNode[],
  pathways: KGNode[],
  supportingPaths: KGPath[],
}
  ▼
LLM returns Pick[] = { conditionId, rationale }
  ▼
Mechanism[] = picks
  .map(p => ({ ...candidatesByConditionId[p.conditionId], rationale: p.rationale }))
  .filter(present)
  ▼
state.mechanisms (each conforms to MechanismSchema, including rationale)
```

`conditionId` is the SNOMED code rather than the PrimeKG node ID so downstream consumers (`generate-search-strategy`, `find-repurposing-candidates`) can correlate mechanisms back to the original FHIR condition without re-querying the crosswalk.

## Error handling

| Failure | Behavior |
|---|---|
| `state.patientProfile == null` | Return `{ error: "No patient profile available" }`. |
| No active conditions on profile | Return `{ mechanisms: [] }`. |
| All conditions unresolvable via crosswalk | Return `{ mechanisms: [] }`. Log warning with the unresolved SNOMED codes + displays. |
| Some conditions unresolvable | Process the resolvable ones; log unresolved. |
| Neo4j down / Cypher throws | `try/catch` at node boundary; return `{ error: "Failed to query KG: <msg>" }`. Same pattern as `extract-patient-profile`. |
| LLM fails / structured output validation fails | Caught by `withStructuredOutput`; return `{ error: "Failed to rank mechanisms: <msg>" }`. |
| LLM picks a `conditionId` not in candidates | Skip it; log warning. Don't fail. |

No in-node retries — graph-level retry policy handles transients.

## Testing

### Unit (Vitest, no external deps)

- `scripts/build-mondo-crosswalk.test.ts` — synthetic SSSOM TSV + nodes CSV (handful of rows including a `MONDO_grouped` row); assert output structure, prefer-exact-over-close, grouped-row fan-out, that `broadMatch` rows are dropped.
- `apps/agent/src/tools/snomed-mondo.test.ts` — `resolveSnomedCondition` returns the right entry / null for unknown codes.
- `apps/agent/src/tools/kg.test.ts` — `buildMechanismsForConditions` with a mocked `Driver` returning canned record sets. Assert: `Mechanism[]` shape, `supportingPaths` construction (correct nodes + edges + relation labels), unresolved-condition skip behavior, ordering preserved from input.
- `apps/agent/src/prompts/mechanism.test.ts` — snapshot the rendered prompt for a fixed `PatientProfile` + candidates input. Locks the prompt shape across edits.

### Integration (gated, `RUN_KG_TESTS=1`)

- `apps/agent/src/nodes/identify-relevant-mechanisms.integration.test.ts` — boots against a running local Neo4j with PrimeKG loaded. For each archetype patient (Hedy Sauer, Brady Schmidt, Pamela Lesch, Marvin Weissnat):
  - At least one condition resolves via the crosswalk.
  - At least one gene target and at least one pathway are returned.
  - LLM call mocked to a deterministic stub (we're testing wiring, not model quality).

Sanity-check that the breast-cancer archetype surfaces BRCA-adjacent pathways, NSCLC surfaces EGFR/ALK-adjacent, RA surfaces TNF/IL-6, and the comorbid patient picks an oncology-driven mechanism over T2DM/CKD.
